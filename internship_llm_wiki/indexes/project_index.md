# Project Index

本頁用於整理 `wiki/projects/` 中的專案筆記。

`project_index.md` 的重點不是保存完整專案內容，而是提供專案型筆記的導覽入口，幫助快速查詢：

- 目前有哪些專案或 PoC
- 每個專案目前處於哪個階段
- 專案包含哪些任務
- 專案使用哪些工具、流程、資料或公式
- 專案目前是否完成、卡住、待確認或已歸檔
- 專案產出了哪些可複用成果

---

## 1. Index 說明

本 index 對應以下資料夾：

| 類型 | 位置 | 說明 |
|---|---|---|
| 專案筆記 | `wiki/projects/` | 實習期間較大型的工作主題、PoC、原型、Demo、階段性專案與成果整理 |

適合放入 `wiki/projects/` 的內容包括：

| 類型 | 範例 |
|---|---|
| PoC 專案 | 概念驗證、小型可行性測試、工具導入驗證 |
| 原型專案 | Prototype、Demo、內部展示用原型 |
| 開發專案 | 系統功能測試、Agent workflow、RAG 原型、資料處理流程 |
| 文件專案 | 報告產出、wiki 建置、流程文件整理 |
| 資料專案 | Parsing pipeline、資料欄位整理、資料處理流程設計 |
| 研究型專案 | 工具比較、方法評估、導入可行性分析 |

## 6. 卡住或待確認專案

本區塊整理目前無法繼續、需要確認、等待權限或等待外部資訊的專案。

| 專案 | 卡住原因 | 需要確認事項 | 確認對象 | 狀態 |
|---|---|---|---|---|
| `[[project_name]]` |  |  | 使用者 / 主管 / 同事 / 待確認 | blocked / pending / waiting_review |

### 常見卡住原因

| 類型 | 說明 |
|---|---|
| 目標不明 | 專案目標、範圍、驗收標準或預期成果尚未確認 |
| 權限問題 | 缺少系統、資料、工具或檔案存取權限 |
| 環境問題 | 本機環境、套件版本、執行環境或部署環境尚未穩定 |
| 資料問題 | 資料來源、欄位、格式、品質或使用限制尚未確認 |
| 技術問題 | 工具、程式、API、模型或流程尚未能穩定運作 |
| 審查問題 | 需要主管、同事、架構師或其他角色確認後才能繼續 |
| 敏感資訊問題 | 專案內容涉及內部系統、資料或流程，需要抽象化後才能整理 |

### 優先度判斷

| 優先度 | 判斷標準 |

| 中 | 有學習或整理價值，但目前不是主要工作核心 |
| 低 | 輔助性專案背景，短期內不影響主要進度 |

## 11. 專案階段定義

| 階段 | 說明 |
|---|---|
| planning | 專案目標、範圍、需求、資料或工具仍在確認 |

| prototype | 正在建立原型或 Demo 雛形 |

## 13. 維護規則

### 必須維護

- 新增 `wiki/projects/` 筆記後，必須更新 [[#3. 專案總表]]。
- 專案開始進行時，應更新 [[#4. 進行中專案]]。
- 專案完成後，應更新 [[#5. 已完成專案]] 與 [[#9. 專案成果追蹤]]。
- 專案卡住或需要確認時，應更新 [[#6. 卡住或待確認專案]]。
- 專案包含任務時，應更新 [[#7. 專案與任務對應]] 與 [[task_index]]。
- 專案使用到工具、流程、資料或公式時，應同步更新對應 index。
- 專案產生錯誤排除經驗時，應同步更新 [[troubleshooting_index]]。
- 若專案內容需要進入首頁近期重點，應更新 [[home]]。

### 不建議

- 不要把完整專案內容寫在本 index。
- 不要把本頁當成專案筆記本體。
- 不要在本頁保存客戶資料、帳號、密碼、API Key、Token、完整內部系統網址或未遮蔽截圖。
- 不要在本頁保存未抽象化的公司內部審查細節、資料表名稱或系統架構。
- 不要在本頁建立完整 Knowledge Graph；完整關聯應放在正式 `wiki/projects/` 筆記中。
- 不要把每個小待辦都建立成專案；小型工作應放入 `wiki/tasks/`。
